# Ribat Watch — 日本のモスク × 政治的区割り

日本全国のモスク・祈祷室・建設予定地（計311地点）を、政治的な区割りの上に重ねて
表示するインタラクティブ地図（Leaflet 製）。モスク密度のヒートマップと、
**自治体 / 選挙区 / 両方 / 非表示** の切替を備える（画面 UI は日本語）。

## 起動方法

このアプリはローカルの GeoJSON を `fetch` で読み込むため、簡易ウェブサーバが必要
（`index.html` を `file://` で直接開くとブラウザに読み込みをブロックされる）。

```bash
# 方法A — ダブルクリック（macOS）:
open serve.command            # ポート8777でサーバを立ててブラウザを開く

# 方法B — 手動（どのOSでも）:
cd Ribat_Watch                # クローン／展開したフォルダへ移動
python3 -m http.server 8777   # 簡易サーバを起動
# ブラウザで http://localhost:8777/app/ を開く
```

> **ビルド不要。** `app/data/` に生成済みの地図データを同梱しているので、
> クローンしてサーバを起動すればそのまま動く（Python やデータの再生成は任意）。

## できること

| 操作 | 選択肢 |
|---|---|
| **区分レベル** | 自治体・選挙区・**両方**・非表示 |
| **行政区の単位** | 市区町村（1,895）・都道府県（47） |
| **選挙区の単位** | 衆院小選挙区（**289 — 現行の2022年改定後の地図**）・参院選挙区（45） |
| **集計指標** | モスクのみ・モスク＋祈祷室・全て（予定地含む） |
| **地点** | モスク（268）・祈祷室（36）・予定地（7）・🔥点ヒートマップ |

**Both** では、行政区を塗り分け（コロプレス）で、選挙区を破線の輪郭で重ねて表示する。
塗り分けの色はレイヤーごとの分位スケール（レベルや指標を変えると再計算）。区にカーソルを
合わせると内訳が表示され、クリックすると件数のポップアップが出る。

## 個人情報の取り扱い（公開版）

元となった有志リストは公開情報だが、公開リポジトリに載せるにあたり、モスクの説明文から
**個人レベルの情報を除去**している（モスク名・所在地・公式サイト・ニュース・Wikipedia・
モスク名義の公式SNSは残す）。除去対象:

- 個人名義の SNS ハンドル（個人の TikTok / Instagram / Facebook アカウント等）
- 特定個人の氏名
- 会衆の国籍・民族に関する記述や、係争的・主観的な論評

除去処理は `parse_kml.py` の `scrub_description()` が担う。除去語の一覧はそれ自体が
個人情報のため公開せず、`scripts/scrub_rules.json`（`.gitignore` 済み）に置く。
未加工の生KML（`mosques_raw.kml`）も個人情報を含むためリポジトリには含めない。

## 処理の流れ（方法論）

1. **取得（スクレイピング）** — マイマップの `mid` を KML エンドポイント
   （`/maps/d/kml?mid=…&forcekml=1`）で書き出し、`parse_kml.py` が3レイヤー・311地点を
   `mosques.geojson` / `mosques.csv` に整形。都道府県 ID コード・`★`削除印・出典 URL・
   説明文も保持する。
2. **現行の衆院小選挙区地図** — 有名な GitHub 上の2つの GeoJSON は**改定前（2022年より前）**の
   地図（検証済み：東京が25区で、30区ではない）。`build_hr.py` は CC0 の CSIS シェープファイルを
   `kucode` ごとにディゾルブして現行の289区を作り、版（東京30・神奈川20・宮城5・広島6）を検証する。
3. **集計** — `build_districts.py` が311地点を、それを含む区に割り当てる（点在判定は
   **フル解像度**の形状で実施）。その後にウェブ用へ簡略化する。都道府県は市区町村から
   ディゾルブし、参院45区は合区2組（鳥取＋島根、徳島＋高知）をまとめて件数を合算する。

## 既知の制限

- **市区町村境界の版は2021年**。2021年以降の一部の市町村合併は未反映。現行版は583MBの
  国土数値情報ダウンロードが必要で、密度地図には過剰なため採用していない。
  都道府県・衆院・参院の線には影響しない。
- **311地点のうち2地点**（お台場の祈祷室と東京ビッグサイトの祈祷室）は、簡略化した海岸線が
  内側に寄る東京湾の埋立地上にあるため、どの市区町村ポリゴンにも入らない。市区町村・都道府県
  レベルでは未集計（いずれもモスクではなく祈祷室）。全311地点は衆院小選挙区には収まる。
- モスクリストは有志が維持する**スナップショット**であり、公式の登録簿ではない。
  `★` は既に Google マップから削除された地点を示す。

## 境界データの再取得（アプリの実行には不要）

`data/raw/` の中身（`muni.geojson`、`senkyoku2022.zip` とその展開先 `senkyoku2022/*.shp`）は
「生の境界データ」で、`build_hr.py` / `build_districts.py` が `app/data/*.geojson` を作るための
**入力**にすぎない。**地図を表示するだけなら不要**（`app/data/` の GeoJSON が生成済みで完結する）。
`.shp` は180MB・ZIPは128MBと大きく、GitHub の100MBファイル上限も超えるため、
リポジトリには含めていない（`.gitignore` 済み）。

出典（いずれも自由に利用可能・境界データはパブリックドメイン/オープンデータ）:

| ファイル | 取得元 | ライセンス |
|---|---|---|
| `muni.geojson`（市区町村） | `https://raw.githubusercontent.com/smartnews-smri/japan-topography/main/data/municipality/geojson/s0001/N03-21_210101.json` | 国土数値情報 N03（要クレジット） |
| `senkyoku2022.zip`（衆院小選挙区） | `https://gtfs-gis.jp/senkyoku2022/senkyoku2022.zip` | パブリックドメイン（CC0） |

境界データを一から作り直す手順:

```bash
# 0) 仮想環境（初回のみ）
python3 -m venv .venv && ./.venv/bin/pip install shapely pyshp

# 1) 生データを取得（上記2ファイルをダウンロード＆ZIP展開。既にあればスキップ）
bash scripts/fetch_raw.sh

# 2) 生成
./.venv/bin/python parse_kml.py                # モスク: KML → GeoJSON/CSV
./.venv/bin/python scripts/build_hr.py         # 衆院小選挙区289を生成
./.venv/bin/python scripts/build_districts.py  # app/data/* を再生成
```

（モスクデータを一から作り直すには、`mosques_raw.kml` をマイマップの KML エクスポートで
取得し、任意で `scripts/scrub_rules.json` を用意した上で `parse_kml.py` を実行する。）
